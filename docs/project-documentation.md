# Glosc Chat 项目文档

## 1. 项目概述

Glosc Chat 是一款面向 Android 和 iOS 的 AI 会话应用。用户可以自行配置兼容的 API 地址、模型和 API Key，在一个统一的移动端界面中使用不同模型进行对话。

项目参考 open-webui 的产品思路：统一模型入口、清晰的会话管理、可扩展的知识库和工具能力、丰富的设置体系。但 Glosc Chat 的优先级不同：它不是以自托管多人 Web 平台为核心，而是以移动端个人使用、本地配置和隐私可控为核心。

## 3. 产品定位

### 3.1 一句话定位

一个移动端优先的 BYOK 多模型 AI 聊天客户端。

### 3.2 核心价值

- 用户掌控 API Key、模型来源和数据存储位置。
- 一个应用中统一管理多个模型供应商。
- 保持移动端会话体验轻量、稳定、可恢复。
- 后续可逐步支持知识库、提示词、工具和跨设备同步。

### 3.3 目标用户

| 用户类型 | 需求 |
| --- | --- |
| AI 重度用户 | 在手机上快速切换模型，保留上下文和历史记录 |
| 开发者/技术用户 | 使用 OpenAI-compatible API、本地模型网关或自建代理 |
| 内容创作者 | 管理常用提示词、长文本会话和知识资料 |
| 企业或团队中的个人用户 | 在不依赖平台账号体系的情况下使用内部模型 API |

## 4. 产品原则

1. 本地优先：会话、配置和密钥优先保存在本机，远程同步作为可选能力。
2. BYOK 优先：用户自己提供 API Key，应用不默认托管用户密钥。
3. 多模型统一：供应商差异由适配层消化，界面提供一致体验。
4. 移动端优先：输入、阅读、历史检索、设置和错误恢复都按手机使用场景设计。
5. 渐进增强：先完成稳定聊天，再扩展知识库、工具、插件和同步。
6. 可解释失败：网络、鉴权、限流、模型不支持等错误应给出可操作反馈。

## 5. 功能范围

### 5.1 已实现

- Vite + Vue 3 前端工程。
- Tauri 2 应用壳。
- 基础 Rust 命令调用示例。
- 基础构建脚本：`dev`、`build`、`preview`、`tauri`。

### 5.2 MVP 范围

MVP 的目标是让用户可以稳定完成一次真实 AI 对话，并能在下次打开应用时继续使用。

| 模块 | MVP 要求 |
| --- | --- |
| 首次启动 | 引导用户添加第一个 API Provider |
| 模型配置 | 支持 OpenAI-compatible API 地址、API Key、模型名 |
| 聊天 | 支持新建会话、发送消息、流式输出、停止生成、重试 |
| 历史记录 | 本地保存会话列表、消息和模型信息 |
| Markdown | 支持代码块、列表、链接、复制 |
| 设置 | 支持编辑 Provider、默认模型、上下文长度、温度等参数 |
| 错误处理 | 展示鉴权失败、网络失败、限流、模型不存在等状态 |
| 数据安全 | 密钥不写入明文 localStorage；发布前引入系统安全存储 |

### 5.3 V1 范围

| 模块 | V1 要求 |

| 移动端体验 | 安全区适配、键盘适配、触控反馈、深浅色主题 |

| 模块 | 规划方向 |

## 6. 非目标

短期内不做以下事情：

- 不建设 open-webui 式完整多人管理后台。
- 不默认托管用户 API Key。
- 不自带模型推理服务。
- 不在 MVP 中实现复杂插件市场。
- 不在 MVP 中实现完整 RAG 管线和向量数据库。

## 7. 核心用户流程

### 7.1 首次配置模型

1. 用户首次打开应用。
2. 应用检测没有 Provider 配置。
3. 展示添加 Provider 表单。
4. 用户输入名称、API Base URL、API Key、模型名。
5. 应用执行轻量连通性测试。
6. 测试通过后保存配置。
7. 用户进入聊天界面。

### 7.2 发起聊天

1. 用户点击新建会话。
2. 选择模型或使用默认模型。
3. 输入消息并发送。
4. 应用创建用户消息和助手占位消息。
5. Provider 适配层发送请求并接收流式响应。
6. UI 增量渲染助手回复。
7. 完成后写入本地历史记录。

### 7.3 恢复历史会话

1. 用户打开应用。
2. 应用读取本地会话索引。
3. 用户从会话列表选择历史会话。
4. 应用加载消息和当时使用的模型参数。
5. 用户继续提问，应用携带必要上下文继续请求。

### 7.4 处理失败

1. 请求失败后，应用保留用户输入和上下文。
2. 错误提示说明失败类型。
3. 用户可以重试、切换模型、编辑配置或复制错误信息。
4. 失败不会破坏已有会话历史。

## 8. 信息架构

MVP 推荐信息架构：

- `聊天`
  - 当前会话
  - 会话列表
  - 模型选择
- `模型`
  - Provider 列表
  - 模型配置
  - 连通性测试
- `提示词`
  - 系统提示词
  - 常用模板
- `设置`
  - 通用设置
  - 数据与隐私
  - 导入导出
  - 关于

移动端可采用底部导航或侧边抽屉。桌面端可采用左侧会话栏 + 中间聊天区 + 右侧设置面板。

## 9. 技术范围

| 层级 | 当前选型 | 说明 |
| --- | --- | --- |
| UI | Vue 3 | 组合式 API，适合构建响应式界面 |
| 类型系统 | TypeScript | 当前开启严格模式 |
| 构建 | Vite | 当前固定端口 `1420` 以适配 Tauri |
| 应用壳 | Tauri 2 | 支持桌面端，后续可配置移动端 |
| 原生层 | Rust | 适合实现安全存储、文件、系统能力和网络代理 |
| 包管理 | 当前 Tauri 配置使用 `yarn` | 仓库暂无 lockfile，建议尽快统一 |

发布前需要确认 `src-tauri/tauri.conf.json` 中的应用标识符。当前值为 `con.gloscai.glosc-chat`，如果是正式域名反写，通常应核对是否为 `com.gloscai.glosc-chat`。

## 10. 数据与隐私要求

- API Key 必须进入系统安全存储或加密存储，不应明文保存到 localStorage。
- 会话内容默认只保存在本机。
- 导出数据前提示用户导出内容可能包含敏感信息。
- 日志不得输出 API Key、完整请求头或用户隐私内容。
- 后续同步能力必须明确加密边界和服务端可见数据范围。

## 11. 质量要求

| 维度 | 要求 |
| --- | --- |
| 稳定性 | 网络中断、应用切后台、流式响应失败后可恢复 |
| 性能 | 长会话滚动不卡顿，消息增量渲染不阻塞输入 |
| 可访问性 | 支持系统字号、深浅色、键盘与读屏基础语义 |
| 可维护性 | Provider、存储、UI 状态分层，避免请求逻辑散落在组件中 |
| 可测试性 | 供应商适配器、消息序列化、错误映射和存储迁移可单元测试 |

## 13. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| API Key 明文存储 | 用户隐私和资金风险 | 使用系统安全存储；日志脱敏 |
| 多供应商接口差异 | 适配复杂度上升 | 先定义统一 Provider 接口；OpenAI-compatible 优先 |
| 移动端网络限制 | 流式输出和后台恢复不稳定 | 抽象请求层；记录请求状态；支持重试 |
| 长会话性能下降 | 聊天界面卡顿 | 虚拟列表、摘要上下文、分段加载 |
| Tauri 移动端经验不足 | 发布周期不确定 | 先保留桌面可运行闭环，再逐步配置移动端 |
| 文档和实现漂移 | 协作成本上升 | 功能变更必须更新 docs 状态 |
